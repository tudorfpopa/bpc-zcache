────────────────────────────────────────────────────────────────────────────────
GOAL
────────────────────────────────────────────────────────────────────────────────
Combine ZCache's skew-associative tag placement with BPC compression and a
buddy-allocated data store to maximise effective LLC capacity while minimising
conflict misses.  The three new subsystems are:

  1. DecoupledDataStore  — buddy allocator for compressed data blocks
  2. FragReservoir       — LFSR best-of-4 reservoir for fragmentation evictions
  3. ZCacheGlue          — coordinator that connects all subsystems on every fill


────────────────────────────────────────────────────────────────────────────────
NEWLY ADDED FILES
────────────────────────────────────────────────────────────────────────────────

glue/decoupled_data_store.hh
glue/decoupled_data_store.cc
  Buddy allocator SimObject.
  - Pool divided into 8-byte segments; supports orders 0–3 (8/16/32/64 B blocks)
  - allocate(sizeBytes) → segment index or ALLOC_FAIL (UINT32_MAX)
  - deallocate(ptr, size) with buddy coalescing
  - Pool initialised as free list of 64 B blocks
  - write(ptr, data, len) stores raw bytes into the pool

glue/frag_reservoir.hh
glue/frag_reservoir.cc
  Pseudo-random fragmentation-eviction reservoir SimObject.
  - 16-bit Fibonacci LFSR (poly 0xB400, seed 0xACE1) samples a random block
    every sampleIntervalTicks ticks via a self-rescheduling gem5 Event
  - Maintains 4 bins × 4 slots (size classes 8/16/32/64 B), keeps coldest
    candidates (lowest lastTouchTimestamp)
  - getVictim(sizeBytes) walks bins from the matching class upward, returns
    the coldest valid candidate in O(1), or nullptr if all bins are empty
  - Uses SimObject (not ClockedObject) with Tick-based interval to avoid
    needing a clock domain when parented under ZCacheGlue

glue/zcache_tags_new.hh
glue/zcache_tags_new.cc
  ZCacheTags subclass that adds per-block compressed-data metadata.
  - Parallel uint32_t arrays blkDataPtr[] and blkAllocSize[] sized to match
    the tag array; resized in tagsInit() after calling the parent's tagsInit()
  - setBlockDataPtr / getBlockDataPtr / getBlockAllocSize / getBlockLastTouch
    accessors used by ZCacheGlue and FragReservoir
  - getNumBlocks() / getBlockByIndex(idx) for LFSR-based random sampling
  - forceInvalidate(blk): invalidates a block bypassing the normal cuckoo walk
    (calls BaseTags::invalidate + zcacheRP->invalidate directly)
  - doMoveBlock override: migrates dataPtr/allocSize metadata before delegating
    to the parent, keeping compressed pointers consistent during cuckoo chains
  - setGlue(ZCacheGlue*): called by ZCacheGlue::startup() to wire the callback
  - insertBlock override: calls glue->handleFill(pkt, blk) when pkt->hasData()

glue/zcache_glue.hh
glue/zcache_glue.cc
  Coordinator SimObject (ZCacheGlue).
  Fill flow in handleFill(pkt, blk):
    1. Build uint64_t chunk vector from pkt data
    2. Call BPC::compress() → get compressedBits
    3. computePaddedSize() → bits→bytes (BPC already pads to power-of-2)
    4. DecoupledDataStore::allocate(paddedSize)
    5. Fragmentation-eviction loop while ALLOC_FAIL:
         - FragReservoir::getVictim(paddedSize)
         - ZCacheTagsNew::forceInvalidate(victim)
         - DecoupledDataStore::deallocate(victim ptr, victim size)
         - Retry allocate
    6. DecoupledDataStore::write(ptr, rawData, paddedSize)
    7. ZCacheTagsNew::setBlockDataPtr(blk, ptr, paddedSize)
  startup(): calls zcacheTags->setGlue(this) and reservoir->setTagsRef(zcacheTags)
  Stats group (GlueStats):
    statFragEvictions            — reservoir-eviction events
    statFillsWithData            — fills carrying data
    statTotalPaddedBytes         — sum of padded sizes across all fills
    statTotalRawBytes            — sum of raw (BPC-reported) sizes across all fills
    statReservoirRequests        — getVictim() calls
    statReservoirHits            — times reservoir returned a valid candidate
    statAvgInternalFragmentation — (paddedBytes - rawBytes) / paddedBytes
    statReservoirHitRate         — reservoirHits / reservoirRequests

SESSION 2 — BUG FIXES
────────────────────────────────────────────────────────────────────────────────

BUG 1: Pool slot leak on normal ZCache evictions
─────────────────────────────────────────────────
Root cause:
  The standard eviction path in gem5 is:
    BaseCache::handleEvictions → BaseTags::invalidate (virtual)
  ZCacheTagsNew did NOT override invalidate(), so the original ZCacheTags::
  invalidate path executed — which never freed the compressed pool slot.
  When cuckoo chains ran doMoveBlock() on an evicted tag slot, the old
  blkDataPtr[dst] was silently overwritten, leaking the compressed pool
  segment forever.  Under a 640 KiB sweep (Scenario C), the pool exhausted
  within the first pass, triggering ALLOC_FAIL on every subsequent fill.

Fix (glue/zcache_tags_new.hh and glue/zcache_tags_new.cc):
  Added ZCacheTagsNew::invalidate() override.
  Before calling BaseSetAssoc::invalidate(blk), the override:
    1. Reads blkDataPtr[idx] and blkAllocSize[idx]
    2. If ptr != ALLOC_FAIL and size > 0 → calls glue->deallocatePoolSlot(ptr, size)
    3. Zeroes blkDataPtr[idx] = ALLOC_FAIL and blkAllocSize[idx] = 0
  This ensures every eviction — whether by normal LRU/ZCache replacement or
  by FragReservoir force-eviction — properly frees its compressed pool slot.

  New declaration in zcache_tags_new.hh:
    void invalidate(CacheBlk *blk) override;

  New implementation in zcache_tags_new.cc:
    Added #include "mem/cache/tags/base_set_assoc.hh"
    void ZCacheTagsNew::invalidate(CacheBlk *blk) {
        uint32_t idx  = getBlockIndex(blk);
        uint32_t ptr  = blkDataPtr[idx];
        uint32_t size = blkAllocSize[idx];
        if (glue && ptr != DecoupledDataStore::ALLOC_FAIL && size > 0)
            glue->deallocatePoolSlot(ptr, size);
        blkDataPtr[idx]   = DecoupledDataStore::ALLOC_FAIL;
        blkAllocSize[idx] = 0;
        BaseSetAssoc::invalidate(blk);
    }

Supporting change in glue/zcache_glue.hh:
  Added public method declaration:
    void deallocatePoolSlot(uint32_t ptr, uint32_t size);

Supporting change in glue/zcache_glue.cc:
  Added implementation (one-liner forwarding to the data store):
    void ZCacheGlue::deallocatePoolSlot(uint32_t ptr, uint32_t size) {
        dataStore->deallocate(ptr, size);
    }


BUG 2: Stale FragReservoir entry invalidating a freshly inserted block
───────────────────────────────────────────────────────────────────────
Root cause:
  After Bug 1 was fixed, the pool could still exhaust temporarily during
  a burst of fills.  The fragmentation-eviction loop calls
  FragReservoir::getVictim(), which can return entries sampled before the
  current tick.  If the sampled block was already invalidated (by a prior
  eviction or cuckoo chain) and then re-inserted (blkDataPtr = ALLOC_FAIL
  from the new fill's insertBlock before handleFill completed), the loop
  called forceInvalidate() on the freshly inserted block, destroying it.
  gem5 then asserted blk->isValid() on a subsequent access to that address.

Fix (glue/zcache_glue.cc — handleFill fragmentation-eviction loop):
  Added a validity guard before acting on each reservoir candidate:

    CacheBlk *victim = static_cast<CacheBlk *>(victimEntry);
    uint32_t victimPtr  = zcacheTags->getBlockDataPtr(victim);
    uint32_t victimSize = zcacheTags->getBlockAllocSize(victim);

    // Guard: skip stale entries — block was invalidated or re-inserted
    if (!victim->isValid()
            || victimPtr == DecoupledDataStore::ALLOC_FAIL
            || victimSize == 0) {
        ptr = dataStore->allocate(paddedSize);  // retry alloc; may now succeed
        continue;
    }

    glueStats.statReservoirHits++;
    glueStats.statFragEvictions++;
    zcacheTags->forceInvalidate(victim);
    dataStore->deallocate(victimPtr, victimSize);
    ptr = dataStore->allocate(paddedSize);

  Stale entries are silently skipped; the guard uses three independent checks
  so no valid block with an ALLOC_FAIL pointer (a bug state) is also skipped.

functional_verify.py  (repo root)
──────────────────────────────────
  gem5 simulation config for three-scenario functional verification.
  CLI: gem5.opt --outdir=m5out_verify_X functional_verify.py --scenario X
       where X in {A, B, C}

  Scenario table:
    A — Normal CDZCache
          pool_bytes="256KiB", sweep=320KiB, period=100 ticks, sim_ticks=50M
          Pool never exhausted; exercises BPC granularity path.
    B — Small Pool / Fragmentation
          pool_bytes="16KiB" (1024x16B = 25% of cache), sweep=320KiB,
          period=100 ticks, sim_ticks=50M
          Forces ALLOC_FAIL for ~75% of fills; exercises reservoir eviction path.
          NOTE: pool_bytes must be "16KiB", not "64KiB" — with 64KiB the pool
          is exactly balanced (each normal eviction frees one slot as each fill
          claims one), so ALLOC_FAIL never occurs.
    C — ZCache Cuckoo Stress
          pool_bytes="256KiB", sweep=640KiB (2.5x cache), period=15000 ticks,
          sim_ticks=200M
          Heavy oversubscription forces many cuckoo chains; validates doMoveBlock
          metadata migration.
          NOTE: period must be > miss latency (~12000 ticks) to serialise misses.
          The class-level pendingSwapChain in ZCacheTags is single-instance;
          concurrent misses with PERIOD=100 would overwrite it before insertBlock
          consumes it.  PERIOD=15000 ensures serial execution.

  Key implementation notes carried forward from cdzcache_test.py:
    - ZCacheTagsNew: all params explicit (size, assoc, block_size, entry_size,
      tag_latency, warmup_percentage=0, sequential_access=False) — Parent.xxx
      proxies cannot resolve when parent SimObject is ZCacheGlue, not a Cache.
    - ZCacheGlue: block_size NOT explicitly set; resolved via Parent.cache_line_size
      proxy from System.
    - mem_ranges = [AddrRange("512MB")] — large enough for the 640 KiB sweep.
    - Two traffic phases (warm_up + measure) each running sim_ticks ticks, followed
      by an exit generator.
    - Output directory anchored via m5.core.setOutputDir() inside the script.
